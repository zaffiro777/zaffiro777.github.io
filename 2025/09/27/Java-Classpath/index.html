ClassPath

​ As we all know, the Java class can exist in 2 forms, the source code and the byte code.

​ Source code requires programmers code manually, e.g the following piece of code is the source code of the PetHome class which is a generic class.

​ ByteCode can be acquired through the compiler. Run the following command, then we get the bytecode of PetHome class.

​ Note that the PetHome class, provided for the Pet Store is not the main class.

​ Then, suppose we develop a simple Dog class, then the bytecode can be acquired in the same way.

Then, create a main class named Demo in the same directory ,which relies on the PetHome class

​ When compile the Demo class, the following error occurred, indicating the PetHome cannot be found by the compiler.

​ In this case, the classpath should be clarified to the compiler.

​ The error before has been eliminated but we get another error that the compiler cannot find the Dog class, which implies the compiler successfully find the bytecode of the PetHome class through the classpath we provided.

	So, we can conclude that one of the functions of the classpath is telling compiler where to find the dependencies.

	What is confusing is why the compiler cannot find the dog class, which is in the same directory with the PetHome class ?

The answer to it is simple. If the classpath is clarified manually , then the compiler will only search the dependencies in the directory provided by classpath, which indicates that the current directory of PetHome class should also be added to the classpath option.

​ The classpath option naturally support multiple paths, which can be separated by the semicolon (or the colon in Linux/MacOS)。

​ Eventually, the compilation is successful.

​

​ Try to run the bytecode. Unfortunately, we still get an error cause the Java Launcher take the same strategy that if the classpath is not specified , it will only search the current directory as the Java compiler, which obviously lead to failure cause there is not PetHome is the current directory.

​ The solution is also the same. Just specify the classpath. After this example, another function of the classpath is obviously telling the Java Launcher where to find the dependencies.

​ This kind of configuring classpath completely maually is quite inconvenient so the classpath is ofen managed by the IDE (Intellij IDEA、Eclipse) in the actual project.

​ Totally speaking, the 2 functions of the classpath is

  • telling Java compiler where to find the byte code on which it relies.
  • telling JVM where to find the byte code on which it relies.
